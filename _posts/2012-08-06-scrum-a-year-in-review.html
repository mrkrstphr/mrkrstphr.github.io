'Scrum: A Year in Review' categories: - Development tags: - agile - scrum status: draft type: post published: false meta: _edit_last: '2' --- Roughly a year ago, a coworker when to the Certified ScrumMaster Workshop and learned all about the Scrum methodology. As soon as he got back, we began doing scrum on our newest project. We organized a scrum team, developed a backlog and started working 2 week sprints. It went fairly well: we were adaptive, we did just-in-time requirements rather than defining the entire project upfront, and we got the project done, satisfying our users. A couple weeks ago, I went to the Certified ScrumMaster Workshop myself, and very quickly realized we were doing quite a bit wrong.

No Definition of Done

We simply didn't have a definition of done, which is a set of rules that define when a user story can be closed and is used to help generate tasks for a user story.

No Scrum Board

We never used a Scrum Board. Initially, we used an Excel spreadsheet that we could all access and update. I, as the ScrumMaster, normally updated the spreadsheet every morning during the daily standup. Eventually, we moved to using Agilo on top of Trac. Using software certainly makes it seem easier and more convenient to use, but it has several drawbacks that we didn't really see.
